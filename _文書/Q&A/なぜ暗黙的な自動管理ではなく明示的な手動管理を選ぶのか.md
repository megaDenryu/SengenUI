# なぜ暗黙的な自動管理ではなく明示的な手動管理を選ぶのか

## 疑問

SengenUIの頻出パターン集では、リスト管理、リソースクリーンアップ、アニメーション等で「明示的に管理する」アプローチを採用している。他フレームワークではこれらを自動化する仕組みがある。なぜ明示的な方を選ぶのか。

## 共通する設計判断

全てのパターンに共通する判断基準: **コードを読んだだけで何が起きるか分かるか？**

自動化された仕組みは便利だが、「何が・いつ・なぜ起きるか」がコードから読み取れなくなる。バグが起きたとき、自動化の裏側を理解していないとデバッグできない。明示的なコードは冗長だが、動作が予測可能。

---

## リスト管理: なぜMap<ID, View>か

フレームワークによるリスト差分検出（配列を渡すと自動で追加・削除・並び替えを判定）は、内部でO(n)のdiffアルゴリズムが走る。1件追加するだけでも全件を走査する。

Map<ID, View>では:
- 追加: `map.set(id, view)` + `container.child(view)` — O(1)
- 削除: `map.get(id).delete()` + `map.delete(id)` — O(1)
- 「何を追加/削除したか」がコードに明示されている。フレームワークが推測するのではない

さらに、フレームワークのdiffではIDの選定ルール（配列indexを使うなetc.）を「知っていないといけない」。Map<ID, View>ではIDはMapのキーとして型で保証される。

## クリーンアップ: なぜ明示的disposeか

自動クリーンアップ（GCやフレームワークのライフサイクルフック）に頼ると:
- **タイマーやWebSocketはGCでは回収されない。** `setInterval` で登録したコールバックはクロージャがタイマーに参照されているため、コンポーネントがDOMから外れてもGCされない
- **「いつクリーンアップが走るか」がフレームワーク依存。** 再レンダリング時か、unmount時か、次のeffect実行時か — これを知っていないとリークする
- **何をクリーンアップするかがコードから見えにくい。** クロージャにキャプチャされた変数が暗黙的にクリーンアップ対象になる

`I破棄可能` + 明示的 `dispose()` では:
- `dispose()` メソッド内に「何を解放するか」が全て書かれている
- 呼び出し責任がOrchestratorに明確にある
- `implements I破棄可能` で型レベルで「このコンポーネントにはdisposeが必要」と宣言される

## アニメーション: なぜdata-attribute + CSS transitionか

JSベースのアニメーションライブラリ（Framer Motion等）は:
- フレームワークのレンダリングサイクルに結合する
- アニメーション状態がJSの変数に散在し、CSSと二重管理になる
- ライブラリのAPI（`motion.div`, `useSpring` 等）を覚える必要がある

data-attribute + CSS transitionでは:
- 状態は `setAttribute()` で変える。アニメーションはCSSが担当。責務が分離されている
- CSSファイルを見れば全てのアニメーション定義が分かる
- 追加ライブラリ不要。ブラウザのCSS transitionエンジンがそのまま使える
- data-attribute状態定数（SengenUIガイド 第13条）と自然に統合される

## 状態管理: なぜauto-trackingではなく明示的購読か

auto-tracking（Signal系の自動依存追跡）は:
- `computed(() => a() + b())` — 何に依存しているかは関数本体を読まないと分からない
- 条件分岐で依存が動的に変わる: `computed(() => flag() ? a() : b())` — flagによって依存先が変わるが、これが暗黙的に処理される
- effectの実行タイミングが依存グラフに基づいて自動決定される。デバッグ時に「なぜこのeffectが走ったか」を追うのが困難

明示的購読（モデルセル + 派生セル）では:
- `派生セル(計算, [ソースA, ソースB])` — ソースが引数として明示されている
- `モデルセル.購読する(fn)` — 何に購読しているかが呼び出し行として見える
- 購読を `_購読を配線する()` メソッドに集約すれば、全ての「状態→View」接続が1箇所でgrep可能

詳細は [宣言的状態管理とは何か](./宣言的状態管理とは何か.md) を参照。

## まとめ

| 手法 | 自動化のアプローチ | 明示的アプローチ | 明示的を選ぶ理由 |
|---|---|---|---|
| リスト管理 | 配列diff + key | Map<ID, View> | O(1)操作、IDが型で保証 |
| クリーンアップ | GC / ライフサイクルフック | I破棄可能 + dispose() | 何を解放するか明示、型で表明 |
| アニメーション | JSアニメーションライブラリ | data-attribute + CSS transition | 責務分離、追加依存なし |
| 状態管理 | auto-tracking Signal | モデルセル + 明示的購読 | 依存が見える、デバッグ可能 |
