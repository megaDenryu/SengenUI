# 「宣言的状態管理」とは何か — Reactのどや顔を分解する

## 疑問

Reactコミュニティは「宣言的な状態管理」を誇らしげに語る。これは本当に優れた設計なのか。SengenUIのエコシステム（Layer 3: SengenUI非依存のTS Utility）にどう取り入れるべきか。

## 「宣言的状態管理」の正体

Reactが言う「宣言的状態管理」を分解すると、3つの独立した概念に分かれる:

1. **状態遷移を純粋関数で書く**（Reducerパターン）
2. **状態を不変値として扱う**（イミュータブル）
3. **状態とUIの対応を宣言する**（UIは状態の関数）

このうち 1 と 2 は**ただの関数型プログラミング**であり、Reactの功績ではない。TypeScriptのDiscriminated Unions + 純粋関数で普通にできる。

3 はReactが再レンダリングで実現しているが、SengenUIでは明示的な購読（モデルセル）で実現する。

つまり「宣言的状態管理」の本質は**React固有の技術ではなく、言語機能だけで達成可能**。

## 本当に価値がある部分

### 価値1: Reducer — 状態遷移を純粋関数に閉じ込める

Reducerの核心: `(現在の状態, アクション) => 新しい状態` という純粋関数。

```typescript
// アクションをDiscriminated Unionで定義
type カート操作 =
    | { readonly kind: "商品追加"; readonly 商品: 商品 }
    | { readonly kind: "商品削除"; readonly 商品ID: 商品ID }
    | { readonly kind: "数量変更"; readonly 商品ID: 商品ID; readonly 数量: number }
    | { readonly kind: "クリア" };

// 状態型
interface カート状態 {
    readonly 商品一覧: ReadonlyArray<カート内商品>;
    readonly 合計金額: number;
}

// Reducer: 純粋関数。副作用なし。テスト容易
function カートリデューサー(状態: カート状態, 操作: カート操作): カート状態 {
    switch (操作.kind) {
        case "商品追加": {
            const 新一覧 = [...状態.商品一覧, { 商品: 操作.商品, 数量: 1 }];
            return { 商品一覧: 新一覧, 合計金額: 合計を計算する(新一覧) };
        }
        case "商品削除": {
            const 新一覧 = 状態.商品一覧.filter(item => item.商品.id !== 操作.商品ID);
            return { 商品一覧: 新一覧, 合計金額: 合計を計算する(新一覧) };
        }
        case "数量変更": {
            const 新一覧 = 状態.商品一覧.map(item =>
                item.商品.id === 操作.商品ID ? { ...item, 数量: 操作.数量 } : item
            );
            return { 商品一覧: 新一覧, 合計金額: 合計を計算する(新一覧) };
        }
        case "クリア":
            return { 商品一覧: [], 合計金額: 0 };
    }
}
```

**Reactは関係ない。** これはただのTypeScript関数。UIフレームワーク不要でテストできる:

```typescript
// テスト: 純粋関数なのでUIなしで検証可能
const 初期状態: カート状態 = { 商品一覧: [], 合計金額: 0 };
const 追加後 = カートリデューサー(初期状態, { kind: "商品追加", 商品: テスト商品 });
assert(追加後.商品一覧.length === 1);
assert(追加後.合計金額 === テスト商品.価格);
```

### 価値2: 状態マシン — 不可能な状態を型で排除する

状態遷移の方向を制約し、「あり得ない状態」をコンパイル時に防ぐ。

```typescript
// 状態の全パターンを型で定義
type 通信状態<T> =
    | { readonly kind: "待機中" }
    | { readonly kind: "通信中"; readonly 中断する: () => void }
    | { readonly kind: "成功"; readonly データ: T }
    | { readonly kind: "失敗"; readonly エラー: Error; readonly 再試行: () => void };

// 有効な遷移だけを許可する関数群
function 通信を開始する<T>(
    リクエスト: () => Promise<T>,
    on状態変更: (状態: 通信状態<T>) => void,
): void {
    const controller = new AbortController();

    on状態変更({
        kind: "通信中",
        中断する: () => controller.abort(),
    });

    リクエスト()
        .then(データ => on状態変更({ kind: "成功", データ }))
        .catch(エラー => on状態変更({
            kind: "失敗",
            エラー,
            再試行: () => 通信を開始する(リクエスト, on状態変更),
        }));
}
```

使う側は `switch (状態.kind)` でexhaustive checkが効く。

### 価値3: 派生状態 — ソースから計算される値

あるデータが別のデータから自動的に導出される場合。

**Reactの問題**: `useMemo` で表現するが、依存配列を手動管理する。間違えてもコンパイルエラーにならない。

**auto-tracking Signalの問題**: 依存が暗黙的。`computed(() => a() + b())` が何に依存するかは関数本体を読まないと分からない。

**提案: 明示的な派生セル**

```typescript
/**
 * 派生セル: ソースのモデルセル群から値を導出する。
 *
 * auto-trackingとの違い:
 * - ソースを明示的に列挙する（コンストラクタ引数）
 * - 何に依存しているかがコード上で見える
 * - 依存の変更忘れは型では防げないが、ソース一覧がコード1箇所にまとまるので見落としにくい
 */
class 派生セル<T> {
    private readonly _内部セル: モデルセル<T>;
    private readonly _購読解除一覧: I購読解除[] = [];

    constructor(
        private readonly _計算する: () => T,
        ソース一覧: モデルセル<unknown>[],
    ) {
        this._内部セル = new モデルセル<T>(this._計算する());
        for (const ソース of ソース一覧) {
            this._購読解除一覧.push(
                ソース.購読する(() => {
                    this._内部セル.設定する(this._計算する());
                })
            );
        }
    }

    get 値(): T { return this._内部セル.値; }

    購読する(fn: (値: T) => void): I購読解除 {
        return this._内部セル.購読する(fn);
    }

    dispose(): void {
        for (const 購読 of this._購読解除一覧) 購読.解除();
    }
}
```

使用例:

```typescript
const 商品一覧 = new モデルセル<カート内商品[]>([]);
const 税率 = new モデルセル<number>(0.1);

// ソースを明示的に列挙。auto-trackingではない
const 合計金額 = new 派生セル(
    () => 商品一覧.値.reduce((sum, item) => sum + item.価格 * item.数量, 0),
    [商品一覧],  // ← 依存が見える
);

const 税込金額 = new 派生セル(
    () => 合計金額.値 * (1 + 税率.値),
    [合計金額, 税率],  // ← 依存が見える
);

// Viewへの購読
税込金額.購読する((金額) => {
    this._合計表示.setTextContent(`${金額}円`);
});
```

## Reactの「宣言的」が実は宣言的でない部分

### useReducer — Reducerは良いが、useReducerが悪い

```jsx
// React
const [state, dispatch] = useReducer(reducer, initialState);
// dispatchの型は Dispatch<Action> だが、
// 「今どの状態からどのアクションが有効か」は型で表現されない
dispatch({ type: "SAVE" });  // 通信中にSAVEを呼べてしまう
```

Reducer関数自体は良い設計。問題はReactの `useReducer` が「今の状態で有効なアクション」を制約できないこと。Discriminated Unionの `switch` では不正な遷移を無視できるが、呼び出し側で不正なdispatchを**型レベルで防げない**。

状態マシンパターンでは、遷移関数の型シグネチャ自体で有効な遷移を制約できる:

```typescript
// 「通信中」状態からのみ「中断」が可能 — 型で保証
function 中断する(状態: Extract<通信状態<unknown>, { kind: "通信中" }>): 通信状態<unknown> {
    状態.中断する();
    return { kind: "待機中" };
}
```

### useState — 「宣言的」に見えて手続き的

```jsx
function Form() {
    const [name, setName] = useState("");
    const [email, setEmail] = useState("");
    const [isValid, setIsValid] = useState(false);

    useEffect(() => {
        setIsValid(name.length > 0 && email.includes("@"));
    }, [name, email]);
    // ↑ isValidはnameとemailから派生するのに、独立したstateとして宣言されている
    //   useEffectで手続き的に同期している。これは「宣言的」ではない
}
```

本当に宣言的なら `isValid` は派生値として宣言される:

```typescript
// 宣言的: isValidはname/emailから決定論的に導出される
const name = new モデルセル("");
const email = new モデルセル("");

const isValid = new 派生セル(
    () => name.値.length > 0 && email.値.includes("@"),
    [name, email],
);
// useEffectによる手続き的同期は不要
```

### Redux — 「予測可能な状態コンテナ」の矛盾

Reduxの売り文句は「予測可能な状態管理」だが:

1. **全状態がグローバルストアにフラットに置かれる**: DDDの集約境界を無視。何でもアクセスできるので、結合度が高くなる
2. **アクションは文字列型ベース**: `dispatch({ type: "ADD_TODO" })` — typoしても型エラーにならない（TypeScript対応で改善されたが、根本設計が文字列前提）
3. **MiddlewareによるAction interceptが暗黙的**: redux-saga, redux-thunkなどのミドルウェアがdispatchを横取りする。どこでどう処理されるかがコードから追跡困難

SengenUIのエコシステムでは:
- Discriminated Unionで型安全なアクション（文字列ベースにしない）
- ドメイン集約ごとに状態を管理（グローバルストアにしない）
- コンストラクタ注入で依存を明示（ミドルウェアで横取りしない）

## 「宣言的状態管理」をLayer 3ユーティリティとして整理

### コンポーネント一覧

| ユーティリティ | 責務 | React相当 | 違い |
|---|---|---|---|
| `モデルセル<T>` | 単一値の保持と変更通知 | `useState` | 明示的購読。再レンダリングなし |
| `派生セル<T>` | 他のセルから値を導出 | `useMemo` | ソースを明示的に列挙。依存配列の罠なし |
| `Reducer関数` | 状態遷移を純粋関数に閉じ込める | `useReducer` | ただのTS関数。フレームワーク不要 |
| `状態マシン` | 有効な遷移を型で制約 | なし（Reactにはない） | TypeScriptのDUで自然に表現 |
| `コマンドリポジトリ` | Undo/Redo | なし（外部ライブラリ） | 既にコードベースに実装あり |

### Reducerパターンの使いどころ

全ての状態にReducerを使う必要はない。判断基準:

1. **状態遷移のパターンが3つ以上ある** → Reducer。遷移ロジックを純粋関数に隔離する価値がある
2. **状態遷移のテストが重要** → Reducer。UIなしで遷移を検証できる
3. **Undo/Redoが必要** → Reducer + コマンドリポジトリ。純粋関数なので逆遷移の定義が容易
4. **単純なon/offや値の更新** → モデルセルで十分。Reducerは過剰

### 状態マシンの使いどころ

1. **「不可能な状態」が存在する** → 状態マシン。型で排除する
   - 例: 「通信中なのにデータがある」「エラーなのにローディング中」はあり得ない
2. **状態遷移の方向に制約がある** → 状態マシン。不正な遷移を防ぐ
   - 例: 「待機中→通信中→成功/失敗」であり、「成功→通信中」は不正
3. **UIの表示切り替えが状態に直結する** → 状態マシン + data-attribute

## Reducerとモデルセルを組み合わせる

Reducer（純粋関数）とモデルセル（変更通知）は直交する概念なので、組み合わせられる:

```typescript
/**
 * Reducerで遷移ロジックを管理しつつ、
 * モデルセルで変更通知を行うStore。
 */
class ReducerStore<TState, TAction> {
    private readonly _状態: モデルセル<TState>;

    constructor(
        初期状態: TState,
        private readonly _reducer: (状態: TState, アクション: TAction) => TState,
    ) {
        this._状態 = new モデルセル<TState>(初期状態);
    }

    get 状態(): TState { return this._状態.値; }

    dispatch(アクション: TAction): void {
        const 新状態 = this._reducer(this._状態.値, アクション);
        this._状態.設定する(新状態);
    }

    購読する(fn: (状態: TState) => void): I購読解除 {
        return this._状態.購読する(fn);
    }
}
```

使用例:

```typescript
// Reducerは純粋関数（テスト容易）
const store = new ReducerStore(初期カート状態, カートリデューサー);

// Viewへの購読（明示的）
store.購読する((状態) => {
    this._商品リストView.全件更新する(状態.商品一覧);
    this._合計View.setTextContent(`${状態.合計金額}円`);
});

// アクション発行
store.dispatch({ kind: "商品追加", 商品 });
```

### これとReduxの違い

| 観点 | Redux | ReducerStore |
|---|---|---|
| スコープ | グローバルシングルトン | ドメイン集約ごとにインスタンス生成 |
| アクションの型 | 文字列ベース（`type: "ADD_TODO"`） | Discriminated Union（`kind: "商品追加"`） |
| ミドルウェア | あり（暗黙的intercept） | なし（副作用はStore外で明示的に処理） |
| 依存注入 | Provider + useSelector（暗黙的Context） | コンストラクタ注入（明示的） |
| UI依存 | react-redux必須 | SengenUI非依存。どのUIフレームワークとも組める |

## まとめ

「宣言的状態管理」の実態は3つの独立した概念。いずれもReact固有ではなく、標準TypeScriptで達成可能:

| 概念 | やること | ツール |
|---|---|---|
| 状態遷移の純粋関数化 | `(state, action) => newState` | Reducer関数（ただのTS関数） |
| 不可能な状態の型による排除 | 遷移の方向を制約 | Discriminated Union + 状態マシン |
| 状態とUIの対応を宣言する | 状態変更時にViewを更新 | モデルセル + 明示的購読 |

Reactの`useState` + `useEffect` による「宣言的」状態管理は、実際には手続き的同期のシンタックスシュガーにすぎない。派生値を独立stateとして持ち `useEffect` で同期するのは、宣言的の対極。

本当に宣言的なのは: 派生セルのようにソースと計算式を宣言し、同期は仕組みに任せること。ただしauto-trackingではなくソースを明示的に列挙する。

## SengenUIアプリケーションでの推奨アプローチ

ここまでの議論を踏まえ、SengenUIのView構築と組み合わせたときに状態管理をどうするかの具体的な指針を示す。

### 全体像: 誰が何を持つか

```
┌─────────────────────────────────────────────────┐
│ Orchestrator（配線係）                            │
│                                                   │
│  ┌──────────┐   購読   ┌──────────────────┐      │
│  │ View群    │ ←─────── │ 状態（Layer 3）   │      │
│  │ (SengenUI)│          │                  │      │
│  │           │ イベント  │ モデルセル        │      │
│  │           │ ───────→ │ 派生セル          │      │
│  │           │          │ ReducerStore     │      │
│  │           │          │ 状態マシン        │      │
│  └──────────┘          └──────────────────┘      │
│        ↑                        ↑                 │
│        │ createComponentRoot    │ コンストラクタ注入 │
│        │                        │                 │
└─────────────────────────────────────────────────┘
```

- **Viewは状態を持たない。** View関数は純粋関数。LV2 View部品は外部からデータを受けて表示する。状態は全てLayer 3のユーティリティが持つ
- **状態はOrchestratorにコンストラクタ注入される。** グローバルストアにしない
- **OrchestratorがViewと状態を配線する。** 購読の登録とイベントハンドラの接続を担う

### 判断フロー: どのツールを使うか

```
状態を管理したい
│
├─ 単一の値で、変わったら知りたい？
│   → モデルセル<T>
│
├─ 別の状態から計算で導出される？
│   → 派生セル<T>（ソースを明示列挙）
│
├─ 状態遷移のパターンが3つ以上ある？
│   │
│   ├─ 「不可能な状態」を排除したい？
│   │   → Discriminated Union による状態マシン
│   │
│   └─ 遷移ロジックをテストしたい？
│       → Reducer関数（純粋関数）
│           └─ 変更通知も必要？→ ReducerStore<TState, TAction>
│
├─ Undo/Redoが必要？
│   → コマンドリポジトリ + Reducer
│
└─ 単純なon/offやフラグ？
    → モデルセルで十分。過剰な抽象化を避ける
```

### 具体例: ポーズ編集画面の状態管理

VoiroStudioのポーズ編集を例に、全体の配線を示す。

#### Step 1: 状態の定義（Layer 3 — SengenUI非依存）

```typescript
// パーツ状態.ts — 純粋なTypeScript。UIフレームワーク非依存

/** 個別パーツの状態 */
class パーツ状態 {
    readonly オンオフ = new モデルセル<OnOff>("off");
    readonly つまみ = new モデルセル<"開" | "無" | "閉">("無");
}

/** ポーズ編集画面全体の状態 */
class ポーズ編集状態 {
    readonly パーツ群: ReadonlyMap<パーツ差分名, パーツ状態>;
    readonly 現在のポーズ名 = new モデルセル<string>("init");
    readonly 保存状態 = new モデルセル<通信状態<void>>({ kind: "待機中" });

    /** 全パーツのon/offが変わったかの派生値 */
    readonly 変更あり: 派生セル<boolean>;

    constructor(パーツ名一覧: パーツ差分名[]) {
        const マップ = new Map<パーツ差分名, パーツ状態>();
        for (const 名前 of パーツ名一覧) {
            マップ.set(名前, new パーツ状態());
        }
        this.パーツ群 = マップ;

        // 全パーツのオンオフセルをソースとして「変更あり」を派生
        const 全オンオフセル = [...マップ.values()].map(p => p.オンオフ);
        this.変更あり = new 派生セル(
            () => 全オンオフセル.some(セル => セル.値 !== "off"),
            全オンオフセル,
        );
    }
}
```

#### Step 2: ロジック（Layer 3 — SengenUI非依存）

```typescript
// ポーズ操作サービス.ts — ドメインロジック。UIフレームワーク非依存

class ポーズ操作サービス {
    constructor(
        private readonly _状態: ポーズ編集状態,
        private readonly _送信機能: ポーズ情報送信機能,
    ) {}

    パーツを切り替える(名前: パーツ差分名, onOff: OnOff): void {
        const パーツ = this._状態.パーツ群.get(名前);
        if (!パーツ) return;
        パーツ.オンオフ.設定する(onOff);
        // → モデルセルが購読者に通知 → Viewが更新される
    }

    async ポーズを保存する(ポーズ名: string): Promise<void> {
        this._状態.保存状態.設定する({ kind: "通信中", 中断する: () => {} });
        try {
            const ポーズデータ = this._現在のポーズデータを構築する();
            await this._送信機能.ポーズ情報送信(ポーズ名, ポーズデータ);
            this._状態.保存状態.設定する({ kind: "成功", データ: undefined });
        } catch (e) {
            this._状態.保存状態.設定する({
                kind: "失敗",
                エラー: e instanceof Error ? e : new Error(String(e)),
                再試行: () => this.ポーズを保存する(ポーズ名),
            });
        }
    }

    private _現在のポーズデータを構築する(): PoseInfo { /* ... */ }
}
```

#### Step 3: Viewの配線（Layer 4 — SengenUIを使用）

```typescript
// ポーズ編集Orchestrator.ts — ここだけがSengenUIに依存

export class ポーズ編集Orchestrator extends LV2HtmlComponentBase implements I破棄可能 {
    protected _componentRoot: DivC;
    private readonly _購読一覧: I購読解除[] = [];

    constructor(
        private readonly _状態: ポーズ編集状態,
        private readonly _サービス: ポーズ操作サービス,
    ) {
        super();
        this._componentRoot = this.createComponentRoot();
        this._購読を配線する();
    }

    protected createComponentRoot(): DivC {
        return (
            div({ class: styles.root }).childs([
                this._ヘッダーを構築する(),
                this._パーツ一覧を構築する()])
        );
    }

    /** 状態 → View の購読を配線する */
    private _購読を配線する(): void {
        // 各パーツの状態変更をViewに反映
        for (const [名前, パーツ] of this._状態.パーツ群) {
            const view = this._パーツViewマップ.get(名前);
            if (!view) continue;

            this._購読一覧.push(
                パーツ.オンオフ.購読する((値) => view.switchBody(値 === "on")),
                パーツ.つまみ.購読する((値) => view.パチパクぴょこつまみ見た目を変更(値)),
            );
        }

        // 保存状態をUIに反映
        this._購読一覧.push(
            this._状態.保存状態.購読する((状態) => {
                switch (状態.kind) {
                    case "通信中":
                        this._保存ボタン.setAttribute(読込状態.attribute, 読込状態.value.読込中);
                        break;
                    case "成功":
                        this._保存ボタン.setAttribute(読込状態.attribute, 読込状態.value.成功);
                        break;
                    case "失敗":
                        this._保存ボタン.setAttribute(読込状態.attribute, 読込状態.value.失敗);
                        break;
                }
            }),
        );

        // 「変更あり」派生値で保存ボタンの有効/無効を切り替え
        this._購読一覧.push(
            this._状態.変更あり.購読する((あり) => {
                this._保存ボタン.setAttributeIf({
                    If: あり,
                    True: { attr: "data-has-changes", value: "true" },
                    False: { attr: "data-has-changes", value: "false" },
                });
            }),
        );
    }

    /** View → 状態 のイベントを配線する */
    private _ヘッダーを構築する(): DivC {
        return (
            div({ class: styles.header }).childs([
                button({ text: "保存" })
                    .bind(btn => { this._保存ボタン = btn; })
                    .onClick(() => {
                        const ポーズ名 = this._状態.現在のポーズ名.値;
                        this._サービス.ポーズを保存する(ポーズ名);
                    })])
        );
    }

    private _パーツ一覧を構築する(): DivC {
        const コンテナ = div({ class: styles.パーツ一覧 });
        this._パーツViewマップ = new Map();

        for (const [名前, _パーツ] of this._状態.パーツ群) {
            const view = new パーツスイッチ({
                パーツ差分名: 名前,
                onパーツ切り替え: (n, onOff) => this._サービス.パーツを切り替える(n, onOff),
                onつまみ変更: (n, 状態) => { /* ... */ },
            });
            this._パーツViewマップ.set(名前, view);
            コンテナ.child(view);
        }
        return コンテナ;
    }

    dispose(): void {
        for (const 購読 of this._購読一覧) 購読.解除();
    }
}
```

### このアーキテクチャの利点

1. **テスト可能性**: `ポーズ編集状態` と `ポーズ操作サービス` はSengenUI非依存なので、DOMなしで単体テスト可能

    ```typescript
    // テスト: UIなしでロジックを検証
    const 状態 = new ポーズ編集状態(["頭", "体", "腕"]);
    const サービス = new ポーズ操作サービス(状態, mockSender);
    サービス.パーツを切り替える("頭", "on");
    assert(状態.パーツ群.get("頭")!.オンオフ.値 === "on");
    assert(状態.変更あり.値 === true);
    ```

2. **依存方向が一方通行**: 状態(Layer 3) ← Orchestrator(Layer 4) → View(Layer 1)。状態がViewを知ることはない

3. **配線が1箇所に集約**: `_購読を配線する()` メソッドを見れば、全ての「状態→View」接続が分かる。散在しない

4. **破棄が明示的**: `dispose()` で全購読を解除。リークしない

5. **フレームワーク非依存**: SengenUIをやめても、状態管理層はそのまま使える
